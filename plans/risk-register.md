## Document Purpose

This register tracks known risks that may impact system stability, data quality, user experience, or business objectives. Each risk includes assessment, mitigation strategies, detection methods, and current status. This document serves as living record of lessons learned and proactive risk management.

## Risk Assessment Framework

### Risk Levels

| Level | Definition | Response Time |
|--------|-------------|----------------|
| **Critical** | Could cause complete service disruption, data loss, or security breach | Immediate (within 24 hours) |
| **High** | Could significantly impact user experience or require manual intervention | Urgent (within 1 week) |
| **Medium** | Moderate impact, recoverable with monitoring | Planned (within 1 month) |
| **Low** | Minor impact, easily recoverable | Monitored (as needed) |

### Probability Scale

| Probability | Definition |
|-------------|-------------|
| **Very High** | > 70% chance of occurrence |
| **High** | 50-70% chance of occurrence |
| **Medium** | 30-50% chance of occurrence |
| **Low** | 10-30% chance of occurrence |
| **Very Low** | < 10% chance of occurrence |

## Active Risks

### External Dependencies

#### RISK-ED-1: Reddit Web Scraping Rate Limiting and Blocking

**Risk Level:** Critical  
**Probability:** Medium  
**Category:** External Dependencies  
**Status:** Active Monitoring

**Description:**
Reddit may rate limit or block web scraping requests if we exceed their limits or violate their terms of service. Since we use web scraping (not the official API), we're more vulnerable to blocking than API-based solutions.

**Impact:**
- Complete service disruption for Reddit-based content ingestion
- User confusion and loss of trust
- Manual intervention required to restore service

**Mitigation Strategies:**
1. **Rate Limiting**: Implement exponential backoff with jitter for requests
2. **Compliance**: Respect robots.txt and rate limit headers
3. **Rotation**: Use rotating user agents and IP addresses (within ToS)
4. **Caching**: Cache responses aggressively to minimize requests
5. **Monitoring**: Monitor request patterns and adjust rate limits dynamically
6. **Circuit Breaker**: Implement circuit breaker pattern for failed requests

**Detection Methods:**
- Track HTTP status codes (429, 403, 503)
- Monitor success rate of scraping requests
- Alert on consecutive failures or pattern changes
- Track response time degradation

**Owner:** Backend Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

#### RISK-ED-2: Subreddit Bans or Deletions

**Risk Level:** High  
**Probability:** Medium  
**Category:** External Dependencies  
**Status:** Active Monitoring

**Description:**
A subreddit we're ingesting from may be banned, set to private, or deleted by Reddit admins. This causes data source loss and user confusion.

**Impact:**
- Data source loss for subscribed users
- User confusion when content disappears
- Manual cleanup required for broken references

**Mitigation Strategies:**
1. **Health Checks**: Implement health checks that verify subreddit accessibility
2. **Graceful Degradation**: Mark source as unavailable, don't crash
3. **User Notification**: Notify users when subscribed subreddit becomes unavailable
4. **Data Archival**: Archive existing data even if source is lost
5. **Fallback**: Suggest alternative subreddits or sources

**Detection Methods:**
- Monitor 403/404 responses from subreddit endpoints
- Check subreddit metadata (subscriber count, description) for changes
- Alert when subreddit status changes
- User reports of missing content

**Owner:** Backend Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

### Data Quality

#### RISK-DQ-1: Low-Quality Content Ingestion

**Risk Level:** Low  
**Probability:** High  
**Category:** Data Quality  
**Status:** Active Monitoring

**Description:**
Ingested content may not meet quality thresholds (low resolution, spam, off-topic). This degrades user experience and wastes storage.

**Impact:**
- Poor user experience with low-quality content
- Increased storage costs for unwanted content
- User dissatisfaction and churn

**Mitigation Strategies:**
1. **Ingest Rules**: Implement IngestRule filters (minScore, mediaOnly, resolution checks)
2. **Quality Review**: Trash Review interface for manual quality control
3. **Automated Analysis**: Automated content analysis where possible
4. **User Feedback**: Implement user feedback mechanisms for quality
5. **Progressive Filtering**: Start strict, loosen based on user engagement

**Detection Methods:**
- Track rejection rates by rule
- Monitor user engagement metrics (likes, saves, views)
- Review Trash Review queue regularly
- User reports of low-quality content

**Owner:** Backend Team

#### RISK-DQ-2: Duplicate Content

**Risk Level:** Low  
**Probability:** Medium  
**Category:** Data Quality  
**Status:** Active Monitoring

**Description:**
Same image/post may be ingested multiple times from different sources or reposts. This creates redundant data and poor UX.

**Impact:**
- Redundant data storage
- Poor user experience (seeing same content multiple times)
- Wasted storage costs

**Mitigation Strategies:**
1. **URL Deduplication**: Bloom filter for URL deduplication before ingestion
2. **Content Hashing**: Content hash checking for text similarity
3. **Graph Linking**: Link duplicate items in graph for unified view
4. **Unique Constraints**: Unique constraints on Post.id and Media.sha256
5. **Cross-Reference**: Cross-reference checks before ingestion

**Detection Methods:**
- Monitor duplicate detection rates
- Track storage growth vs. unique content
- User reports of duplicate content
- Analyze graph for duplicate nodes

**Owner:** Backend Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

**Detection Methods:**

- Alert on connection failure rates
- Track query timeouts
- Monitor Neo4j Aura dashboard

#### RISK-INF-2: GraphQL API Performance Degradation

**Risk Level:** High  
**Probability:** Medium  
**Category:** Infrastructure  
**Status:** Active Monitoring

**Description:**
Complex GraphQL queries may become slow as data volume grows. N+1 query problems, missing indexes, or inefficient Cypher queries can cause significant performance degradation.

**Impact:**
- Slow frontend, poor user experience
- Increased database load
- Potential service timeouts

**Mitigation Strategies:**
1. **Query Analysis**: Query complexity analysis and limits
2. **Indexes**: Database indexes on frequently queried fields (userId, publishDate, platform)
3. **Pagination**: Cursor-based pagination for all list queries
4. **Caching**: Valkey query result caching (300s TTL for feed queries)
5. **Monitoring**: Performance monitoring and alerting (p95, p99 latency)
6. **Batching**: Query batching and data loader patterns
7. **Optimization**: Neo4j query plan analysis for optimization

**Detection Methods:**
- Track query execution times (p50, p95, p99)
- Monitor Valkey cache hit rates
- GraphQL query complexity metrics
- User-reported slow performance
- Neo4j slow query logging

**Owner:** Backend Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

**Risk Level:** High  
**Probability:** Low

**Description:**
Valkey cache service may fail, causing all queries to hit Neo4j directly. This can cause significant performance degradation and increased database load.

**Impact:**

2. **Health Checks**: Health check endpoints for cache connectivity
3. **Automatic Failover**: Automatic failover to direct Neo4j queries
4. **Cache Warming**: Cache warming on service restart
5. **Monitoring**: Monitor cache memory usage and eviction rates

**Detection Methods:**
- Cache connection health monitoring
- Track cache hit rate (alert if < 70%)
- Monitor Neo4j query load increase
- Alert on cache service unavailability

**Owner:** DevOps Team

### Operational

#### RISK-OP-1: Ingestion Worker Failures

**Risk Level:** High  
**Probability:** Low  
**Category:** Operational  
**Status:** Active Monitoring

**Description:**
Background workers that fetch and ingest content may crash, fail silently, or become unresponsive. This leads to outdated feeds and missed content.

**Impact:**
- Stale data
- Missed content
- User dissatisfaction
- Manual intervention required

**Mitigation Strategies:**
1. **Health Monitoring**: Worker health monitoring with heartbeat mechanism
2. **Auto Restart**: Automatic restart on failure with exponential backoff
3. **Dead Letter Queue**: Dead letter queue for failed jobs with retry logic
4. **Visibility**: Pipeline status endpoint for visibility
5. **Alerting**: Alert on worker downtime or extended sync delays
6. **Graceful Shutdown**: Graceful shutdown handling
7. **Queue Processing**: Queue-based job processing (Valkey queues)

**Detection Methods:**
- Last sync timestamp monitoring per handle
- Worker heartbeat/health checks every 30 seconds
- Alert when sync age exceeds threshold (e.g., 1 hour)
- Track job failure rates and error types
- Monitor queue depth and processing rate

**Owner:** Backend Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

### Security

#### RISK-SEC-1: Authentication Token Compromise

**Risk Level:** Critical  
**Probability:** Low  
**Category:** Security  
**Status:** Active Monitoring

**Description:**
JWT authentication tokens may be compromised through various attack vectors (XSS, CSRF, token leakage). This allows unauthorized access to user data.

**Impact:**
- Unauthorized access to user data
- Data breach
- User trust loss
- Legal liability

**Mitigation Strategies:**
1. **Token Security**: Use strong signing keys (RS256)
2. **Short Lifetime**: Short token lifetime (7 days) with refresh tokens
3. **Secure Storage**: Store tokens securely (httpOnly cookies)
4. **CSRF Protection**: Implement CSRF protection
5. **XSS Prevention**: Implement Content Security Policy
6. **Token Revocation**: Implement token revocation mechanism

**Detection Methods:**
- Monitor for unusual access patterns
- Track login locations and times
- Alert on multiple failed login attempts
- User reports of unauthorized access

**Owner:** Security Team  
**Last Review:** 2025-12-29  
**Next Review:** 2025-01-29

**Probability:** Very Low  
**Category:** Security  
**Status:** Active Monitoring

**Description:**
If end-to-end encryption is implemented, loss of encryption keys would make user data permanently unrecoverable.

**Detection Methods:**
